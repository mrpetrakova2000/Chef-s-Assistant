# Product Proposal: Chef's Assistant

## 1. Обоснование идеи

**Прикладная задача:** Разработка мультиагентной системы, которая по запросу пользователя о желаемом блюде автоматически составляет оптимальный список покупок с сравнением цен в разных интернет-магазинах.

**Актуальность:** В современном мире онлайн-шопинг стал основным каналом покупки продуктов, но фрагментация рынка (множество магазинов с разными ценами, акциями и ассортиментом) создает когнитивную нагрузку на пользователя. Агентный подход позволяет делегировать эту рутинную задачу ИИ, который может одновременно "посетить" несколько магазинов, сравнить предложения и вернуть оптимальный результат.

## 2. Цель и метрики

**Цель проекта:** Разработать прототип мультиагентной системы, способной за < 120 секунд предоставить пользователю оптимальный список покупок для приготовления блюда с сравнением предложений из 2-3 магазинов.

### Метрики успеха:

#### Продуктовые метрики:
- **Время ответа (p95 latency):** ≤ 120 секунд для 95% запросов
- **Success Rate:** ≥ 80% запросов успешно завершаются (найдено ≥ 80% ингредиентов)

#### Агентские/Технические метрики:
- **Точность извлечения ингредиентов:** ≥ 90% совпадение с эталонным рецептом
- **Точность маппинга:** ≥ 85% выбранных товаров соответствуют ингредиенту
- **Coverage:** ≥ 70% ингредиентов успешно маппятся на товары в магазинах
- **Cost per query:** бесплатная квота Mistral API (первично)

## 3. Потенциальные сценарии использования

### Счастливый путь (Happy Path):
**Запрос:** "Хочу оливье на 4 персоны"

**Процесс:**
1. Система определяет кулинарный запрос
2. Генерирует ингредиенты: картофель, морковь, яйца, колбаса, горошек, майонез
3. По каждому ингредиенту ищет в 3 магазинах
4. Сравнивает цены и возвращает смешанный список с лучшими предложениями

### Edge-кейс 1: Синонимы и альтернативы
**Запрос:** "Спагетти карбонара"

**Проблема:** Рецепт требует панчетту, которая может отсутствовать

**Ожидаемое поведение:** 
- Система находит аналоги (бекон/грудинка)
- В ответе помечает: "Заменитель: бекон (вместо панчетты)"

### Edge-кейс 2: Частичная доступность
**Запрос:** "Утка по-пекински"

**Проблема:** Утка может быть только в 1 из 3 магазинов

**Ожидаемое поведение:**
- Система успешно находит утку в одном магазине
- Для остальных ингредиентов выбирает лучшие цены
- В отчете указывает специфику поиска

### Edge-кейс 3: Неоднозначность запроса
**Запрос:** "Приготовь что-нибудь вкусное"

**Ожидаемое поведение:**
- Система запрашивает уточнение: "Какое блюдо вы хотели бы приготовить?"

## 4. Ограничения

### Технические ограничения:
- **p95 latency:** < 120 секунд
- **Availability:** 99% в часы демо
- **Error rate:** < 5% на этапе классификации и планирования

### Операционные ограничения:
- **Бюджет на LLM:** (?)
- **Бюджет на API:** Бесплатные API
- **Storage:** Только оперативная память

## 5. Архитектурный набросок

Система строится на фреймворке LangGraph и включает следующие модули:

**Классификатор (Classifier Agent)** — анализирует входной запрос и определяет, относится ли он к кулинарной тематике. Принимает решение о запуске основного сценария.

**Планировщик (Planner Agent)** — на основе одобренного запроса генерирует структурированный список ингредиентов для указанного блюда, используя LLM.

**Координатор поиска (Search Coordinator Agent)** — управляет основным циклом обработки: последовательно перебирает ингредиенты из списка и контролирует завершение процесса.

**Маршрутизатор (Router Agent)** — для каждого ингредиента определяет категорию товаров в конкретном магазине, учитывая специфику категоризации разных торговых сетей.

**Парсер (Parser Agent)** — выполняет поиск товаров в магазине по указанной категории, используя инструменты веб-скрапинга или API.

**Селектор (Selector Agent)** — анализирует список найденных товаров и выбирает наиболее подходящий для данного ингредиента в рамках одного магазина.

**Компаратор (Multi-Store Comparator Agent)** — сравнивает лучшие предложения из разных магазинов для одного ингредиента и определяет оптимальный вариант.

**Менеджер результатов (Update Results Agent)** — аккумулирует выбранные товары в общем состоянии системы.

**Репортер (Reporter Agent)** — формирует итоговый ответ для пользователя: структурированный список покупок с ценами и ссылками.

**Кэширующий агент (Cache Agent)** — хранит промежуточные результаты парсинга для ускорения повторных запросов.

**Менеджер памяти (Memory Agent)** — ведет историю выполнения сессии для отладки и анализа.

### Интеграции:
- **Внешняя база знаний:** API магазинов, API или LLM для рецептов как источник знаний
- **Инструменты:** Модули для парсинга магазинов, сравнения цен, генерации отчетов
- **State & Memory:** Общее состояние графа, кэш в памяти, история сессии

## 6. Потенциальный data flow

Схема приведена на рис. workflow_1.0.png.\
**Вход:** "Хочу приготовить борщ"

1. **Classifier Agent (LLM):** Определяет, что запрос кулинарный → переход к Planner
2. **Planner Agent (LLM):** Генерирует список ингредиентов: свекла, картофель, капуста, мясо, лук, морковь, томатная паста
3. **Coordinator Agent:** Запускает цикл по ингредиентам и магазинам
4. **Router Agent (LLM):** Для ингредиента "свекла" определяет категорию "Овощи" в текущем магазине
5. **Parser Agent (Инструмент):** Выполняет парсинг категории, получает список товаров
6. **Selector Agent (LLM):** Выбирает лучший товар для свеклы в этом магазине
7. **Компаратор (LLM):** После сбора данных из всех магазинов сравнивает предложения и выбирает лучшее
8. **Update Results Agent:** Сохраняет результат в состоянии
9. **Coordinator Agent:** Переходит к следующему ингредиенту или завершает цикл
10. **Reporter Agent (LLM + Инструмент):** Формирует итоговый отчет для пользователя

### Какие части делегированы LLM/Agent:

| Этап | Делегировано LLM/Agent | Пояснение |
|------|------------------------|-----------|
| **Классификация запроса** | ✅ LLM | LLM определяет интент пользователя по тексту |
| **Генерация списка ингредиентов** | ✅ LLM | LLM извлекает рецепт из своих знаний |
| **Определение категории в магазине** | ✅ LLM | LLM маппит ингредиент на категорию товаров |
| **Выбор лучшего товара в магазине** | ✅ LLM | LLM сравнивает названия и выбирает наиболее релевантный |
| **Сравнение предложений из магазинов** | ✅ LLM | LLM взвешивает цену и качество |
| **Форматирование ответа** | ✅ LLM | LLM структурирует вывод для пользователя |
| **Управление циклом обработки** | ❌ Код | Координатор работает по жесткой логике |
| **Парсинг магазинов** | ❌ Инструмент | Выполняется через HTTP-запросы/скрапинг |
| **Обновление состояния** | ❌ Код | Механическая операция с данными |
| **Кэширование** | ❌ Код | Хранение в памяти по ключам |

### Распределение ответственности между компонентами:

| Компонента | Что делает | Ответственность |
|-----------|------------|------------------|
| **Classifier Agent** | Анализ текста запроса | Логика классификации (LLM) |
| **Planner Agent** | Генерация ингредиентов | Извлечение знаний (LLM) |
| **Coordinator Agent** | Управление очередью | Логика циклов (Код) |
| **Router Agent** | Определение категории | Семантический маппинг (LLM) |
| **Parser Agent** | Сбор данных с сайтов | Веб-скрапинг (Инструмент) |
| **Selector Agent** | Выбор товара в магазине | Ранжирование (LLM) |
| **Comparator Agent** | Межмагазинное сравнение | Оптимизация выбора (LLM) |
| **Update Results Agent** | Сохранение результатов | Работа с состоянием (Код) |
| **Reporter Agent** | Формирование ответа | Генерация текста (LLM) |
| **Cache Agent** | Кэширование данных | Хранение в памяти (Код) |
| **Memory Agent** | Логирование сессии | Запись истории (Код) |

### Обоснование распределения:

**Почему LLM:**
- Задачи требуют семантического понимания (что такое "карбонара", какой продукт ближе к "пармезану")
- Нужна гибкость в обработке синонимов и замен
- Сравнение требует учета нечисловых факторов (бренд, репутация)

**Почему код/инструменты:**
- Парсинг магазинов — механическая задача с четким протоколом
- Управление циклами — детерминированная логика
- Кэширование — требует предсказуемости и скорости
